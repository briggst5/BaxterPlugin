# Training and Development/Standard Work/Code-for-Credit Story Standard Work

#Overview
_Code-for-Credit (CfC)_ is the term given to the risk and test driven process by which a cross-functional scrum team realizes production quality software and associated QMS compliant Design History File (DHF) artifacts at the User Story level.  

The process ensures that cross‑functional team members collaborate tightly during a sprint, emphasizing efficiency, quality, and opportunities for automation.  The work is performed collaboratively across roles as part of a Scrum team.  The activities focus on producing all necessary design, development, and testing outputs expected from the perspective of the User Story.

The minimal cross-functional engagement required to deliver a CfC User Story is listed below:
- Systems
- UX (if the User Story contains scope associated with user facing functionality)
- Software Test
- Software

##NOTES/Discussion:
- CfC tasks should include confirmation that Feature impact assessment is still accurate.  Update with cross functional team as needed.  Driven by Systems.
- To close a CfC story:
   - PO confirms done to story DoD.
   - Process check to confirm we are meeting goals of CfC.
   - Team check to confirm team agrees story is done.

#Standard Work
_Standard Work_  defines the common set of activities performed by the Scrum team during most CfC User Stories. It represents approximately 80% of the recurring work needed to produce design artifacts, code, tests, and supporting DHF documentation at the story level.  While standard work defines a common set of interactions and expectations across the cross-functional scrum team, it is import to note that **_all relevant project planning documents (i.e. PDLM Project Plan, Software Development Plan, Software Verification Plan, etc.) in compliance with all relevant QMS processes (i.e. GQPs, SOPs, etc.) are applicable and ultimately define the deliverables, rigor, and approvals required._**

Teams must exercise engineering judgment to assess the scope of the User Story to determine if all aspects of the standard work apply.  More importantly, teams must also access if the User Story requires *additional* work to be considered complete and QMS compliant beyond what is contained in the Standard Work.

To drive consistency in User Stories intended to deliver CfC, a template CfC User Story with the associated tasks can be found here: #7083
When writing CfC User Stories, teams should copy this template as a starting point for their assessment of the definition of done and the tasks required to achieve it.
